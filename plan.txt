Key design: Health data endpoints do NOT bypass the memory pipeline. They convert structured data into MemoryCandidate objects and inject them into the STM-C buffer. This way, the existing dedup, scoring, conflict resolution, and LTM gate all apply automatically.


# Example: CGM readings → memory candidate  
def cgm_to_candidate(readings: List[CGMReading]) -> MemoryCandidate:  
    avg = sum(r.glucose_mg_dl for r in readings) / len(readings)  
    spikes = [r for r in readings if r.glucose_mg_dl > 180]  
    return MemoryCandidate(  
        text=f"CGM session: avg {avg:.0f} mg/dL, {len(spikes)} spikes detected",  
        category="cgm_pattern",  
        context=f"Time range: {readings[0].timestamp} to {readings[-1].timestamp}",  
    )

Phase 4: CGM Correlation Engine
Goal: Correlate CGM data with food, mood, and activity to generate actionable insights.

SOW alignment: "Real-time CGM-centric intelligence", "Anomaly detection indicators", "Trend graphs and alerts", "Poor correlation between behaviours and outcomes"

File	Purpose

How it works:

When a CGM batch is ingested, the correlation_engine queries recent LTM memories (last 4 hours) for food, mood, and activity entries.
It builds a timeline: [food log at 12:30] → [glucose spike at 13:15] → [mood: stressed at 14:00]
It uses Gemini to generate a correlation insight: "Glucose spiked to 195 mg/dL ~45 min after eating white rice. This pattern has occurred 3 times this week."
The insight is stored as a cgm_pattern memory candidate with high salience.
New LangGraph node: Add a health_correlation node between ltm_gate and chat:

This node only activates when health data is present in the recent context. It's a no-op for regular chat turns.

Architecture: The nudge engine runs as a background task (not in the LangGraph pipeline). It's triggered by:

A periodic scheduler (every 30 min) that checks user state
Real-time CGM data ingestion (immediate alerts)
The Node.js backend calling POST /api/nudges/check
Nudges are stored in PostgresStore under a ("nudges", user_id) namespace and delivered via Firebase FCM through the Node.js backend.

Future-Proofing Design Principles
To handle changes that will inevitably come:

1. Plugin architecture for health modules: Each health module (cgm_processor, nudge_engine, food_recognition, etc.) is a standalone class with a standard interface. New modules can be added without modifying the core pipeline.

2. Feature flags in settings: Add ENABLE_CGM, ENABLE_NUDGES, ENABLE_FOOD_RECOGNITION, ENABLE_VOICE flags. Disabled features return 404 on their endpoints and skip their LangGraph nodes.

3. LangGraph conditional edges: Instead of a fixed linear pipeline, use conditional edges so health nodes only activate when relevant data is present:


# In builder.py — future state  
builder.add_conditional_edges(  
    "ltm_gate",  
    lambda state: "health_correlation" if has_health_data(state) else "chat",  
)
4. Versioned API: All endpoints under /api/v1/. When breaking changes come, add /api/v2/ without breaking existing frontend code.

5. Abstract integration interfaces: All external services (Pinecone, Firebase, Nutritionix, Speech API) are wrapped in abstract classes. Swapping providers means changing one file, not the entire codebase.

6. Separate health memory namespace: Health data memories use distinct categories (cgm_pattern, medication, dietary, etc.) that can be queried independently. If the SOW adds new health signals later (e.g., blood pressure, weight), just add a new category — no schema migration needed.
